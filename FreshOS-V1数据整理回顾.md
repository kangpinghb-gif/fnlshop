# FreshOS V1 数据整理回顾

## 一、回顾结论

当前数据整理方向总体符合总纲预期。

原因：

1. 没有先做复杂AI，而是先做数据闭环。
2. 已围绕库存、销售、到货、盘点修正建立 V1 数据基础。
3. 已明确大表哥、订单Excel、人工盘点三类数据来源。
4. 已把“实时库存、理论库存、人工修正值”的关系定义清楚。
5. 已把字段范围收敛到 V1 必需项，避免系统过重。

当前阶段符合总纲里 V1 的要求：

```text
数据同步
库存修正
订货建议
风险提醒
禁止复杂AI
```

## 二、与总纲逐项对照

### 1. 数据接入层

总纲要求：

- ERP / POS / 外卖 / 人工输入等数据接入
- 优先 API，其次数据库、RPA、Web自动化

当前整理结果：

- 已确定大表哥作为历史经营主数据源。
- 已建立大表哥导出预设。
- 已安装并优化 `dabiaoge` skill。
- 已支持订单Excel作为到货数据来源。
- 已设计人工盘点修正模板。

符合预期。

当前简化是合理的：

```text
V1 不直接接所有系统
先用大表哥导出 + 订单Excel + 人工盘点模板跑通闭环
```

### 2. 库存可信度引擎

总纲要求：

- ERP库存
- 理论库存
- 实际库存
- 库存可信度

当前整理结果：

- 已定义实时库存为大表哥时点库存。
- 已定义理论库存公式。
- 已定义人工盘点数量为修正值。
- 已定义库存数据优先级：

```text
实时库存快照 > 理论库存 > 人工盘点修正值
```

符合预期。

但当前“实际库存”没有直接作为主库存，而是作为人工修正值。这个偏离是合理的，因为你已明确人工盘点不作为主要库存值，只用于纠偏。

### 3. 销量预测引擎

总纲要求：

- 历史销量
- 星期、节假日、天气、活动等因素

当前整理结果：

- 已定义 `sales_daily`。
- 已明确历史销售以大表哥为主。
- 已明确如需日粒度，用大表哥期间趋势/期间推移导出。

符合 V1 预期。

V1 暂时没有天气、节假日、活动字段，这是符合总纲路线的，因为这些属于 V2。

### 4. 智能订货引擎

总纲要求：

- 预测销量
- 库存可信度
- 安全库存
- MOQ
- 包装规格
- 配送周期

当前整理结果：

- 已保留近期日均销量。
- 已保留箱装数。
- 已保留订货批量。
- 已默认每日配送，`delivery_cycle_days = 1`。
- 已保留实时库存和理论库存基础。

基本符合预期。

仍缺：

- 安全库存字段还没有最终落表。
- 订货建议表还没有在数据库设计中单独展开。

建议下一步补充 `order_suggestions` 表。

### 5. 库存生命周期引擎

总纲要求：

- 到货日期
- 批次
- 剩余寿命
- 批次成本

当前整理结果：

- 已明确到货日期来自订单导入。
- 已明确到货日期同时作为生产日期口径。
- 已暂缓复杂批次成本。

部分符合预期。

当前偏简化：

```text
V1 用到货日期 + 保质期计算库存年龄
不做复杂批次成本
```

这是合理的。总纲中也建议 V1 先跑通闭环。

仍缺：

- 没有单独设计轻量批次/到货日库存表。
- 当前 `fresh_order_imports` 能承接到货，但还没有专门沉淀批次库存状态。

建议后续增加轻量表：

```text
inventory_arrival_batches
```

### 6. 库存风险引擎

总纲要求：

- 临期
- 滞销
- 高库存
- 缺货风险
- 高损耗风险

当前整理结果：

- 已具备风险计算所需底层数据：
  - 销售日汇总
  - 实时库存
  - 到货日期
  - 报损数量/金额
  - 盘盈盘亏
  - 人工盘点修正

部分符合预期。

仍缺：

- 风险结果表尚未落入数据库设计。
- 风险评分规则还没有单独拆成可执行规则。

建议下一步补充：

```text
inventory_risks
risk_rule_configs
```

### 7. 动态出清引擎

总纲要求：

- 动销
- 剩余库存
- 剩余寿命
- 时间段
- 折扣建议

当前整理结果：

- 目前只整理了出清所需的底层数据。
- 没有做动态出清表和调价逻辑。

符合 V1 预期。

因为总纲明确动态出清应放在 V3，V1 只做风险提醒。

### 8. 员工执行引擎

总纲要求：

- 自动派任务
- 执行确认
- 延迟追踪

当前整理结果：

- 已明确 V1 暂不维护门店订货负责人。
- 没有做任务系统。

符合 V1 预期。

员工任务系统属于 V3，不应该现在做。

## 四、当前数据整理的优点

### 1. 范围收敛正确

没有把供应商评分、动态定价、员工任务、AI学习提前放进 V1。

### 2. 数据来源现实

没有假设直接接 ERP API，而是先使用：

```text
大表哥导出
订单Excel
人工盘点模板
```

这符合当前实际落地条件。

### 3. 库存口径清楚

已经明确：

```text
实时库存是主库存
理论库存用于校验
人工盘点用于修正
```

这符合总纲“不相信ERP库存”的核心思想。

### 4. 订单口径清楚

已经明确：

```text
到货日期 = 订单导入值
入库数量 = 当日到货值
历史数据 = 大表哥为主
```

这对后续理论库存计算很重要。

### 5. 字段简化合理

已去掉大量 V1 不需要字段，符合“先跑通闭环”的原则。

## 六、是否符合总纲预期

结论：符合。

当前数据整理已经完成了 V1 的底座：

```text
数据接入方案
商品门店基础数据
销售历史口径
库存主口径
订单到货口径
人工盘点修正口径
导入模板
字段映射
导入异常机制
```

但还没有完全覆盖 V1 的业务输出层：

```text
订货建议
库存风险提醒
风险处理状态
```

所以当前阶段适合进入下一步：

```text
补充 order_suggestions 和 inventory_risks 表
  ↓
设计订货建议和风险评分规则
  ↓
继续导出大表哥销售/库存/收货样例
  ↓
开始写数据库 schema
```

## 七、下一步建议

下一步优先补两张表：

1. `order_suggestions`
2. `inventory_risks`

理由：

- 总纲 V1 的核心不是只整理数据，而是输出“订货建议”和“风险提醒”。
- 这两张表是从数据底座走向业务闭环的关键。
