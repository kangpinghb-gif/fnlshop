# FreshOS 生鲜AI经营系统总纲（开发版）

## 一、项目定位

### 项目名称

FreshOS（生鲜AI经营操作系统）

### 项目目标

通过数据驱动与AI辅助，实现：

- 自动订货
- 库存预测
- 动态控损
- 自动出清
- 动态定价
- 门店协同运营

最终目标：

让生鲜门店从“经验经营”升级为“系统经营”。

## 二、核心问题定义

生鲜行业核心问题：

| 问题 | 本质 |
| --- | --- |
| 缺货 | 订少了 |
| 损耗 | 订多了 |
| 毛利下降 | 出清失控 |
| 库存不准 | 数据脏 |
| 店长能力不稳定 | 经验不可复制 |

系统核心目标：

在“断货成本”和“损耗成本”之间动态平衡。

## 三、系统核心理念

### 1. 不相信ERP库存

核心原则：

ERP库存不等于实际库存。

系统必须建立三层库存体系：

- ERP库存
- 理论库存
- 实际库存

### 2. AI不是核心

第一阶段不依赖复杂AI模型。

先建立：

- 数据闭环
- 规则引擎
- 经营逻辑

之后再逐步引入：

- 机器学习
- 动态预测
- AI学习

### 3. 生鲜核心不是销量预测

核心是库存控制。

系统重点：

- 库存年龄
- 动销速度
- 损耗风险
- 生命周期

### 4. 系统必须允许人工干预

生鲜行业存在大量：

- 临时事件
- 本地事件
- 经验判断

系统必须支持人机协同。

### 模块2：库存可信度引擎（核心）

#### 系统目标

解决库存不准问题。

#### 三层库存体系

ERP库存：ERP原始库存。

理论库存：系统自动推算库存。

```text
理论库存 = 昨日库存 + 入库 - 销售 - 预计损耗
```

实际库存：人工抽盘库存。

#### 库存可信度

系统为每个商品生成库存可信度。

| 商品 | 可信度 |
| --- | --- |
| 土豆 | 95% |
| 香菜 | 42% |

#### 系统自动处理

- 库存修正
- 异常识别
- 自动提醒盘点
- 高频误差监控

### 模块3：销量预测引擎

#### 目标

预测未来销量。

#### 影响因素

- 历史销量
- 天气
- 星期
- 节假日
- 活动
- 季节
- 客流

#### 商品分类预测

高频短保类：

- 叶菜
- 草莓
- 鲜奶

特点：

- 高波动
- 高损耗
- 高时效

长保稳定类：

- 土豆
- 洋葱

特点：

- 稳定周转
- 库存水位管理

活动爆品类：

- 榴莲
- 荔枝

重点：活动预测。

### 模块4：智能订货引擎

#### 系统目标

自动生成订货建议。

#### 影响因素

- 预测销量
- 库存可信度
- 当前库存年龄
- 安全库存
- MOQ起订量
- 包装规格
- 配送周期
- 供应商稳定性
- 商品生命周期

#### 订货逻辑

```text
订货量 =
(预测销量 + 安全库存 - 修正库存)
× 损耗修正系数
× 供应商稳定系数
```

#### 必须支持MOQ圆整

示例：

- 香蕉按箱采购
- 一箱20kg

系统必须自动圆整。

### 模块5：库存生命周期引擎

#### 系统目标

管理库存年龄。

#### 核心原则

库存数量不重要，库存年龄更重要。

#### 批次库存管理

必须支持：

- 到货日期
- 批次
- 剩余寿命
- 批次成本

#### 生命周期阶段

| 阶段 | 状态 |
| --- | --- |
| 新鲜期 | 正常销售 |
| 黄金销售期 | 主销售阶段 |
| 风险期 | 动销下降 |
| 临期 | 出清阶段 |

### 模块6：库存风险引擎

#### 系统目标

识别库存风险。

#### 风险类型

- 临期
- 滞销
- 高库存
- 缺货风险
- 高损耗风险

#### 风险评分

系统生成库存风险指数。

| 商品 | 风险 |
| --- | --- |
| 草莓 | 91 |
| 香菜 | 84 |

### 模块7：动态出清引擎（核心盈利模块）

#### 系统目标

#### 动态出清依据

- 销速

- 剩余寿命
- 时间段
- 客流


动态调价：

- 降价
- 满减
- 第二件折扣

防止用户形成固定时间等打折行为。

### 模块8：员工执行引擎

## 十、可实施性检核

### 总体结论

可以实施，而且方向正确。

但实施路径必须收敛：FreshOS不应先做“AI系统”，而应先做“库存经营闭环系统”，AI放在后续阶段逐步增强。

最关键的判断是：

- 生鲜的核心不是单纯销量预测，而是库存可信度、库存年龄、订货和出清之间的联动。
- V1不要追求大而全，要先把每日经营闭环跑通。
- 早期系统应以规则引擎为主，AI作为后续增强能力。

### V1建议定位

FreshOS V1 = 生鲜库存可信度 + 智能订货 + 风险提醒系统。

V1建议只保留以下核心能力：

1. 商品主数据
2. 销售数据导入
3. 库存三账体系
4. 库存可信度评分
5. 订货建议
6. 风险提醒

### V1每日运行闭环

```text
每天早上：
系统给出订货建议

每天营业中：
系统发现缺货、高库存、滞销、临期等风险

每天晚上：
系统复盘销售、损耗、库存偏差
```

### 主要实施风险

#### 1. 数据接入难度会被低估

ERP、POS、外卖平台、报损数据格式通常不统一。

V1不建议同时接所有平台，建议先选一个门店、一个ERP或POS数据源跑通。

#### 2. 库存批次管理很难

如果门店实际没有严格按批次收货、上架、报损，系统里的批次会变成“理论批次”。

V1可以先做简化版：按到货日自动生成批次，用FIFO推算消耗。

#### 3. 动态定价涉及权限和执行

自动改价不是纯技术问题，还涉及收银系统、价签、外卖平台、员工执行和顾客感知。

建议V3再做，V1只做“出清建议”。

#### 4. AI学习需要足够闭环数据

没有“建议、人工修改、执行、结果”的闭环数据，AI学习没有基础。

所以V1必须记录：

- 系统建议值
- 人工修改值
- 修改原因
- 最终执行值
- 后续结果

#### 5. 门店执行是最大变量

系统提醒了，不代表员工执行了。

员工任务系统可以放到V3，但V1至少要有风险处理状态：

- 未处理
- 已确认
- 已忽略
- 已完成

### 推荐优先级

#### 第一优先级

- 商品主数据
- 销售数据
- 库存数据
- 理论库存计算
- 库存可信度
- 订货建议

#### 第二优先级

- 风险评分
- 临期提醒
- 滞销提醒
- 缺货提醒
- 简单经营看板

#### 第三优先级

- 天气与节假日影响
- 动态出清建议
- 员工任务
- 供应商评分

### 最终判断

FreshOS具备现实可行性和商业价值。

但第一阶段必须避免“大而全AI系统”的陷阱，建议先用规则引擎跑通库存经营闭环，再逐步引入预测、动态定价、员工执行和AI学习。
