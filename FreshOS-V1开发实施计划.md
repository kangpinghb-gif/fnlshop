# FreshOS V1 开发实施计划

## 一、当前阶段判断

当前 V1 已经推进到 V1.1，核心口径已经统一为：

- 每天 12:00 生成次日订货建议。
- 使用 0-12 点实时销售做轻量趋势修正。
- 使用预计到货前库存参与订货建议。
- Hermes 负责自动导出 / 抓取大表哥数据并触发 `freshos-worker`。
- 企业微信 / 飞书查看摘要、附件和异常。

当前已完成：

- 产品总纲。
- 多门店 V1 / V1.1 产品需求。
- 数据库与导入设计。
- 大表哥导出字段初步验证。
- 订单导入样本分析。
- 人工盘点修正模板。
- 商品可售天数字段设计。
- 大表哥基础数据导入。
- 大表哥日数据导入。
- 12 点趋势修正预测。
- 预计到货前库存订货口径。

当前重点已经从“等待新数据前的准备工作”转为“云服务器部署 + Hermes 拉齐 + 正式大表哥数据验证”。

## 二、V1.1 实现形态

V1.1 不优先做完整 Web 后台，先做云端自动订货助手。

整体形态：

```text
云服务器 freshos-worker
  ↓
Hermes 定时执行
  ↓
自动抓取/导入大表哥和订单数据
  ↓
FreshOS 计算订货建议和库存风险
  ↓
生成 Excel/CSV 明细
  ↓
企业微信/飞书推送摘要和附件
```

职责划分：

| 模块 | 职责 |
| --- | --- |
| Hermes | 定时调度和执行任务 |
| freshos-worker | 抓取、导入、清洗、匹配、计算、生成文件、推送 |
| PostgreSQL | 存储基础数据、导入数据、计算结果、任务日志 |
| 企业微信/飞书 | 推送每日结果、异常提醒、附件 |
| Excel/CSV | 承载订货建议、库存风险、导入异常明细 |

## 三、V1.1 最小闭环目标

V1.1 最小闭环：

```text
Hermes 定时触发
  ↓
抓取或导入基础数据
  ↓
抓取或导入销售/库存/到货/盘点修正
  ↓
匹配门店和商品
  ↓
计算库存口径
  ↓
计算销量预测
  ↓
生成订货建议
  ↓
输出库存风险
  ↓
生成订货建议和风险明细
  ↓
企业微信/飞书推送
```

验收标准：

- 能自动或半自动获取门店、商品、门店商品关系。
- 能获取最近30天销售数据。
- 能获取库存、损耗、订货/收货数据。
- 能导入订单到货数据。
- 能导入人工盘点修正。
- 能生成门店商品维度的订货建议。
- 能提示负库存、缺货、积压、临期、过期、数据缺失等风险。
- 能生成每日订货建议明细文件。
- 能生成每日库存风险明细文件。
- 能通过企业微信或飞书推送每日结果。

## 五、开发顺序

### 1. 数据库 schema

先把《FreshOS-V1数据库与导入设计》落成数据库结构。

优先创建：

- `stores`
- `products`
- `store_products`
- `sales_daily`
- `inventory_snapshots`
- `inventory_loss_daily`
- `purchase_receipts_daily`
- `fresh_order_imports`
- `stock_count_adjustments`
- `import_exceptions`
- `inventory_positions`
- `inventory_age_batches`
- `sales_forecasts`
- `order_suggestions`
- `inventory_risks`
- `job_runs`

本阶段目标：

- 先建表。
- 先保证字段够用。
- 支持 worker 任务幂等重跑。
- 记录每次任务运行时间、状态、错误信息。
- 不做复杂权限、复杂审批、复杂审计。

### 2. Hermes 任务清单

当前建议任务节奏：

| 时间 | 任务 | 说明 |
| --- | --- | --- |
| 12:00 | 导出 / 抓取大表哥基础数据和实时销售 | 形成半日趋势输入 |
| 12:10 | 导入基础档案和日数据 | 更新门店、商品、销售、库存、到货 |
| 12:20 | 导入订单和盘点修正 | 形成当日到货和修正口径 |
| 12:30 | 计算库存口径、销量预测和到货前库存 | 生成中间结果 |
| 12:40 | 生成订货建议和风险 | 形成最终结果 |
| 12:50 | 导出报表并推送 | 摘要 + 附件 |
| 16:00 | 可选补跑半日趋势校验 | 后续再启用 |

### 3. 导入和抓取数据合同

为每种来源定义固定输入字段和输出表。

需要整理：

- 大表哥商品/门店/门店商品抓取合同
- 大表哥销售日汇总抓取合同
- 大表哥库存/损耗抓取合同
- 大表哥订货/收货抓取合同
- 生鲜订单导入合同
- 人工盘点修正导入合同

本阶段目标：

- 每个抓取/导入来源对应一个解析入口。
- 每个解析入口明确必填字段。
- 无法匹配的数据写入 `import_exceptions`。
- Hermes 抓取失败或字段缺失时，失败原因可追踪。

### 4. 门店商品匹配规则

先实现最小匹配：

```text
门店：优先 store_name，其次 store_code
商品：优先 product_code，其次订单商品名称
```

商品过滤规则：

- 商品名称包含 `折` 的，视为出清 / 特价商品，不计入常规订货建议。
- 商品名称等于 `D系统用代表商品` 的，视为系统代表商品，不计入常规订货建议。
- 上述商品仍可用于销售统计和风险观察，但默认不作为自动订货商品。

异常进入：

- `unmatched_store`
- `store_code_name_conflict`
- `unmatched_product`
- `missing_product_code`

本阶段目标：

- 不追求一次性自动匹配所有商品。
- 先把无法匹配的暴露出来。
- 允许后续人工补商品映射。

### 5. 计算任务

按顺序实现：

```text
inventory_positions
  ↓
inventory_age_batches
  ↓
sales_forecasts
  ↓
order_suggestions
  ↓
inventory_risks
```

V1 计算原则：

- 规则优先，不做复杂 AI。
- 计算结果可解释。
- 每个结果能追溯输入数据。
- 允许数据缺失，但要写风险。

### 6. 报表导出

第一版先输出文件，不先做页面。

每日文件：

- `订货建议明细.xlsx`
- `库存风险明细.xlsx`
- `导入异常明细.xlsx`

订货建议明细至少包含：

- 门店
- 商品
- 当前库存
- 预测销量
- 安全库存
- 已订未到
- 建议订货量
- 订货原因
- 风险标记

库存风险明细至少包含：

- 门店
- 商品
- 风险类型
- 风险等级
- 风险说明
- 相关数量
- 处理状态

### 7. 企业微信/飞书推送

先支持 webhook 推送。

推送内容：

- 每日摘要
- 高风险清单摘要
- 订货建议附件
- 库存风险附件
- 任务失败提醒

消息示例：

```text
FreshOS 订货建议 - 2026-05-25

门店：全部门店
需订商品：128 个
高风险商品：17 个
缺货风险：9 个
临期/积压：8 个

附件：
1. 订货建议明细.xlsx
2. 库存风险明细.xlsx
```

后续再考虑：

- 分门店推送
- 分角色推送
- 消息卡片交互
- 人工确认回写

## 七、工具使用方式

V1.1 第一阶段的工具使用方式详见：

```text
FreshOS-V1工具使用方式与业务操作流程.md
```

核心结论：

- V1.1 不是完整 Web 后台，而是阿里云服务器上的自动订货助手。
- 日常入口是 Hermes：负责自动导出大表哥 40/42 数据，并触发 freshos-worker 任务链。
- systemd timer 只作为阿里云服务器兜底调度方式，不作为大表哥导出的责任方。
- 业务查看入口是企业微信 / 飞书摘要和报表文件。
- 系统维护入口是服务器命令行、任务日志和导入异常明细。
- 人工修改订货建议、盘点修正、商品匹配修复先通过报表和模板完成。
